🎓 什么是 Git 和 GitHub？

1Git - 版本控制系统

Git 是一个分布式版本控制系统，用于跟踪代码的变化历史。它可以：

  • 记录每次代码修改
  • 回退到任意历史版本
  • 多人协作开发
  • 创建分支进行独立开发

2GitHub - 代码托管平台

GitHub 是基于 Git 的在线代码托管平台，提供：

  • 远程仓库存储
  • 团队协作工具
  • 代码审查（Pull Request）
  • 项目管理（Issues、Projects）
  • 持续集成/部署（GitHub Actions）
💡 为什么要学习 GitHub？
  • 全球最大的开源社区
  • 程序员必备技能
  • 展示个人项目和能力
  • 参与开源项目
  • 团队协作必需工具

📋 核心概念

术语 说明
Repository（仓库） 存储项目代码和历史记录的地方
Commit（提交） 保存代码的一个快照，记录某个时间点的状态
Branch（分支） 独立的开发线，可以并行开发不同功能
Merge（合并） 将一个分支的修改合并到另一个分支
Pull Request（PR） 请求将代码合并到主分支的申请
Clone（克隆） 将远程仓库复制到本地
Push（推送） 将本地提交上传到远程仓库
Pull（拉取） 从远程仓库下载最新代码
Fork（派生） 复制别人的仓库到自己账号下

⚙️ 安装 Git

Windows 系统

  1. 访问 https://git-scm.com/download/win
  2. 下载安装包并运行
  3. 安装过程中保持默认选项即可
  4. 安装完成后，右键菜单会出现 "Git Bash"

git --version # 输出类似：git version 2.40.0

🔐 配置 SSH 密钥（推荐）

使用 SSH 密钥可以免密码推送代码，更安全便捷。

1. 生成 SSH 密钥

# 生成新的 SSH 密钥 ssh-keygen -t ed25519 -C "your.email@example.com" # 如果系统不支持 ed25519，使用 RSA ssh-keygen -t rsa -b 4096 -C "your.email@example.com" # 按提示操作： # 1. 按 Enter 使用默认路径 # 2. 输入密码（可选，直接 Enter 跳过） # 3. 再次确认密码

2. 查看公钥

# Windows type %USERPROFILE%\.ssh\id_ed25519.pub # Mac/Linux cat ~/.ssh/id_ed25519.pub # 输出类似： # ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIxxx... your.email@example.com

3. 添加到 GitHub

  1. 登录 GitHub，点击右上角头像
  2. 选择 Settings
  3. 左侧菜单选择 SSH and GPG keys
  4. 点击 New SSH key
  5. Title 填写描述（如 "我的电脑"）
  6. Key 粘贴刚才复制的公钥
  7. 点击 Add SSH key

4. 测试连接

ssh -T git@github.com # 首次连接会提示，输入 yes # 成功输出：Hi username! You've successfully authenticated...

📦 创建新仓库

方法一：在 GitHub 网站创建

  1. 登录 GitHub
  2. 点击右上角 + 号
  3. 选择 New repository
  4. 填写仓库信息：
    • Repository name：仓库名称（必填）
    • Description：项目描述（可选）
    • Public/Private：公开或私有
    • Initialize this repository with：
      • ✅ Add a README file（推荐勾选）
      • Add .gitignore（选择语言模板）
      • Choose a license（选择开源协议）
  5. 点击 Create repository

方法二：本地创建并推送

# 1. 创建项目文件夹 mkdir my-project cd my-project # 2. 初始化 Git 仓库 git init # 3. 创建 README 文件 echo "# 我的项目" > README.md # 4. 添加到暂存区 git add README.md # 5. 提交 git commit -m "初始提交" # 6. 在 GitHub 创建空仓库后，关联远程仓库 git remote add origin https://github.com/你的用户名/仓库名.git # 7. 推送到 GitHub（首次推送） git push -u origin main
💡 提示 -u 参数会设置上游分支，之后只需 git push 即可。

📥 克隆现有仓库

# HTTPS 方式（无需配置 SSH） git clone https://github.com/用户名/仓库名.git # SSH 方式（需配置 SSH 密钥） git clone git@github.com:用户名/仓库名.git # 克隆到指定文件夹 git clone https://github.com/用户名/仓库名.git 自定义文件夹名 # 克隆指定分支 git clone -b 分支名 https://github.com/用户名/仓库名.git

💻 日常开发流程

1. 查看状态

# 查看工作区状态 git status # 简洁模式 git status -s

2. 添加文件到暂存区

# 添加指定文件 git add 文件名.txt # 添加多个文件 git add 文件1.txt 文件2.js # 添加所有修改的文件 git add . # 添加指定目录 git add src/ # 交互式添加 git add -p

3. 提交更改

# 提交暂存区的文件 git commit -m "提交说明" # 添加并提交（跳过 git add） git commit -am "提交说明" # 修改上次提交 git commit --amend -m "新的提交说明" # 打开编辑器编写详细提交信息 git commit
📝 提交信息规范（推荐）
类型(范围): 简短描述 详细描述（可选） 相关 Issue（可选）
常用类型：
  • feat: 新功能
  • fix: 修复 Bug
  • docs: 文档更新
  • style: 代码格式（不影响功能）
  • refactor: 重构代码
  • test: 测试相关
  • chore: 构建工具、依赖更新
  • perf: 性能优化
示例：
  • feat(登录): 添加微信登录功能
  • fix(支付): 修复订单金额计算错误
  • docs: 更新 API 文档

4. 推送到远程

# 推送到远程仓库 git push # 首次推送（设置上游分支） git push -u origin main # 推送到指定分支 git push origin 分支名 # 强制推送（慎用！） git push -f

5. 拉取远程更新

# 拉取并合并 git pull # 拉取指定分支 git pull origin main # 只拉取不合并 git fetch # 拉取所有远程分支 git fetch --all
⚠️ 注意 每次开始工作前，先执行 git pull 获取最新代码，避免冲突！

📊 查看历史

# 查看提交历史 git log # 简洁模式（一行显示） git log --oneline # 图形化显示分支 git log --graph --oneline --all # 查看最近 5 条 git log -5 # 查看某个文件的历史 git log 文件名 # 查看某次提交的详细信息 git show 提交ID # 查看文件修改内容 git diff # 查看暂存区和上次提交的差异 git diff --staged

🔧 分支基本操作

查看分支

# 查看本地分支 git branch # 查看所有分支（包括远程） git branch -a # 查看远程分支 git branch -r # 查看分支详细信息 git branch -v

创建分支

# 创建新分支 git branch 分支名 # 创建并切换到新分支（推荐） git checkout -b 分支名 # 或使用新命令 git switch -c 分支名 # 基于特定提交创建分支 git branch 分支名 提交ID

切换分支

# 切换分支 git checkout 分支名 # 或使用新命令 git switch 分支名 # 切换到上一个分支 git checkout -

合并分支

# 1. 切换到目标分支（如 main） git checkout main # 2. 合并指定分支 git merge 分支名 # 不使用快进模式（保留分支历史） git merge --no-ff 分支名 # 合并时添加提交信息 git merge 分支名 -m "合并说明"

删除分支

# 删除本地分支 git branch -d 分支名 # 强制删除（未合并的分支） git branch -D 分支名 # 删除远程分支 git push origin --delete 分支名 # 清理已删除的远程分支引用 git remote prune origin

🔄 常用分支工作流

Git Flow 工作流（推荐）

1. 开发新功能

# 从 develop 创建功能分支 git checkout develop git pull git checkout -b feature/new-feature # 开发并提交 git add . git commit -m "feat: 添加新功能" # 推送到远程 git push origin feature/new-feature

2. 合并功能

# 切换到 develop git checkout develop git pull # 合并功能分支 git merge --no-ff feature/new-feature # 推送 git push # 删除功能分支 git branch -d feature/new-feature git push origin --delete feature/new-feature

3. 发布版本

# 从 develop 创建发布分支 git checkout -b release/v1.0.0 develop # 修复 Bug、更新版本号等 git commit -am "chore: 准备发布 v1.0.0" # 合并到 main git checkout main git merge --no-ff release/v1.0.0 git tag -a v1.0.0 -m "版本 1.0.0" # 合并回 develop git checkout develop git merge --no-ff release/v1.0.0 # 删除发布分支 git branch -d release/v1.0.0

👥 团队协作

Fork 工作流（开源项目常用）

  1. Fork 仓库：在 GitHub 上点击 Fork 按钮，复制到自己账号
  2. 克隆到本地：
    git clone https://github.com/你的用户名/仓库名.git
  3. 添加上游仓库：
    git remote add upstream https://github.com/原作者/仓库名.git git remote -v # 查看远程仓库
  4. 创建分支并开发：
    git checkout -b feature/my-feature # 修改代码... git add . git commit -m "feat: 添加新功能"
  5. 同步上游更新：
    git fetch upstream git checkout main git merge upstream/main
  6. 推送到自己的仓库：
    git push origin feature/my-feature
  7. 创建 Pull Request：在 GitHub 上点击 "New Pull Request"

🔀 Pull Request（PR）

什么是 Pull Request？

Pull Request 是向项目贡献代码的方式，让维护者审查你的代码后再合并到主分支。

创建 PR 的步骤

  1. 推送分支到远程仓库
  2. 在 GitHub 仓库页面，点击 Pull requests
  3. 点击 New pull request
  4. 选择要合并的分支：
    • base：目标分支（如 main）
    • compare：你的功能分支
  5. 填写 PR 信息：
    • 标题：简洁描述改动
    • 描述：详细说明改动内容、原因、测试情况
  6. 点击 Create pull request

PR 描述模板（推荐）

## 改动说明 简要描述本次改动的内容 ## 改动类型 - [ ] 新功能 - [ ] Bug 修复 - [ ] 文档更新 - [ ] 代码重构 - [ ] 性能优化 ## 相关 Issue Closes #123 ## 测试情况 - [ ] 已通过单元测试 - [ ] 已通过集成测试 - [ ] 已在本地环境测试 ## 截图（如适用） （添加截图） ## 其他说明 （补充信息）

审查 PR

  • 查看代码变更
  • 添加评论和建议
  • 批准（Approve）或请求修改（Request changes）
  • 合并 PR（Merge pull request）
💡 PR 最佳实践
  • 保持 PR 小而专注，一次只解决一个问题
  • 提供清晰的描述和上下文
  • 及时响应审查意见
  • 确保通过所有测试
  • 合并前保持分支最新

🐛 Issues（问题追踪）

创建 Issue

  1. 进入仓库，点击 Issues
  2. 点击 New issue
  3. 填写标题和描述
  4. 添加标签（Label）：bug、enhancement、documentation 等
  5. 分配给相关人员（Assignees）

Issue 模板示例

关闭 Issue

  • 手动关闭：在 Issue 页面点击 "Close issue"
  • 通过提交关闭：在提交信息中使用关键词

🎯 .gitignore 文件

.gitignore 用于指定 Git 应该忽略的文件和目录。

创建 .gitignore

# 在项目根目录创建 touch .gitignore

常用规则

🔄 Stash（暂存工作）

当需要切换分支但又不想提交当前修改时，可以使用 stash。

# 暂存当前修改 git stash # 暂存并添加说明 git stash save "修改说明" # 包括未追踪的文件 git stash -u # 查看 stash 列表 git stash list # 恢复最近的 stash git stash pop # 恢复指定的 stash git stash apply stash@{0} # 删除 stash git stash drop stash@{0} # 清空所有 stash git stash clear # 查看 stash 的内容 git stash show -p stash@{0}

⏮️ 撤销操作

撤销工作区修改

# 撤销单个文件的修改 git checkout -- 文件名 # 或使用新命令 git restore 文件名 # 撤销所有修改 git checkout -- . git restore .

撤销暂存区

# 取消暂存 git reset HEAD 文件名 # 或使用新命令 git restore --staged 文件名 # 取消所有暂存 git reset HEAD

撤销提交

# 撤销最后一次提交，保留修改 git reset --soft HEAD^ # 撤销最后一次提交，修改回到工作区 git reset --mixed HEAD^ git reset HEAD^ # 默认是 mixed # 撤销最后一次提交，丢弃所有修改（危险！） git reset --hard HEAD^ # 撤销最近 3 次提交 git reset --soft HEAD~3 # 回退到指定提交 git reset --hard 提交ID

创建反向提交

# 创建一个新提交来撤销指定提交（推荐） git revert 提交ID # 撤销最近的提交 git revert HEAD # 撤销多个提交 git revert HEAD~3..HEAD
⚠️ reset vs revert
  • reset：重写历史，适合本地未推送的提交
  • revert：创建新提交，适合已推送的提交（推荐）

🔍 搜索和查找

# 在工作区搜索内容 git grep "搜索内容" # 在指定提交中搜索 git grep "搜索内容" 提交ID # 显示行号 git grep -n "搜索内容" # 查找哪次提交引入了某行代码 git blame 文件名 # 查找删除了某个文件的提交 git log --all --full-history -- 文件名 # 查找包含特定内容的提交 git log -S "代码内容" # 查找某个作者的提交 git log --author="作者名" # 查找某个时间段的提交 git log --since="2024-01-01" --until="2024-12-31"

🎨 Git 别名

设置别名可以简化常用命令。

# 设置别名 git config --global alias.st status git config --global alias.co checkout git config --global alias.br branch git config --global alias.ci commit git config --global alias.unstage 'reset HEAD --' git config --global alias.last 'log -1 HEAD' git config --global alias.lg "log --graph --oneline --all" git config --global alias.visual "log --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit" # 使用别名 git st # 等同于 git status git co main # 等同于 git checkout main git lg # 图形化日志

🔗 子模块（Submodules）

子模块允许你在一个 Git 仓库中包含另一个 Git 仓库。

# 添加子模块 git submodule add https://github.com/用户名/仓库名.git 路径 # 克隆包含子模块的仓库 git clone --recursive https://github.com/用户名/仓库名.git # 初始化子模块 git submodule init git submodule update # 更新子模块 git submodule update --remote # 删除子模块 git submodule deinit 路径 git rm 路径

🔄 Rebase（变基）

Rebase 可以让提交历史更加线性和整洁。

# 将当前分支变基到 main git checkout feature-branch git rebase main # 交互式 rebase（整理最近 3 次提交） git rebase -i HEAD~3 # 继续 rebase git rebase --continue # 跳过当前提交 git rebase --skip # 中止 rebase git rebase --abort
⚠️ Rebase 注意事项
  • 不要对已推送到公共仓库的提交进行 rebase
  • Rebase 会改写历史，可能导致冲突
  • 团队协作时优先使用 merge

交互式 Rebase 命令

命令 说明
pick 保留该提交
reword 保留提交，但修改提交信息
edit 保留提交，但停下来修改
squash 合并到前一个提交
fixup 合并到前一个提交，丢弃提交信息
drop 删除该提交

🎯 Cherry-pick（精选提交）

将指定的提交应用到当前分支。

# 应用单个提交 git cherry-pick 提交ID # 应用多个提交 git cherry-pick 提交ID1 提交ID2 # 应用提交范围 git cherry-pick 提交ID1..提交ID2 # 继续 cherry-pick git cherry-pick --continue # 中止 cherry-pick git cherry-pick --abort

🔧 常见问题解决

1. 合并冲突

问题：执行 git pull 或 git merge 时出现冲突

git pull origin main # CONFLICT (content): Merge conflict in 文件名

解决步骤：

  1. 打开冲突文件，查找冲突标记：
    <<<<<<< HEAD 你的代码 ======= 别人的代码 >>>>>>> branch-name
  2. 手动编辑，保留需要的代码，删除冲突标记
  3. 添加并提交：
    git add 文件名 git commit -m "fix: 解决冲突"
💡 使用工具解决冲突
  • VS Code：内置冲突解决工具
  • 命令：git mergetool

2. 误提交敏感信息

问题：不小心提交了密码、密钥等敏感信息

解决方案：

# 如果还没推送，撤销提交 git reset --soft HEAD^ git reset HEAD 敏感文件 # 如果已经推送，从历史中完全删除 git filter-branch --force --index-filter \ "git rm --cached --ignore-unmatch 敏感文件" \ --prune-empty --tag-name-filter cat -- --all # 强制推送 git push origin --force --all # 更简单的方法：使用 BFG Repo-Cleaner # 下载：https://rtyley.github.io/bfg-repo-cleaner/ bfg --delete-files 敏感文件 git reflog expire --expire=now --all git gc --prune=now --aggressive
⚠️ 重要
  • 立即更改泄露的密码/密钥
  • 通知团队成员重新克隆仓库
  • 将敏感文件添加到 .gitignore

3. 推送被拒绝

问题：git push 时提示 "Updates were rejected"

原因：远程仓库有新的提交

解决方案：

# 方法1：先拉取再推送 git pull --rebase git push # 方法2：合并后推送 git pull git push # 方法3：强制推送（危险！会覆盖远程） git push -f

4. 找回删除的提交

问题：误删了提交或分支

解决方案：使用 reflog

# 查看所有操作历史 git reflog # 找到要恢复的提交ID，然后 git checkout 提交ID git checkout -b 恢复的分支名 # 或直接重置到该提交 git reset --hard 提交ID

5. 修改最后一次提交

问题：提交后发现有遗漏或错误

解决方案：

# 修改文件后 git add 修改的文件 git commit --amend # 只修改提交信息 git commit --amend -m "新的提交信息" # 如果已经推送，需要强制推送 git push -f
⚠️ 注意 只对未推送或自己的分支使用 --amend，不要修改公共分支的历史。

6. 大文件问题

问题：推送时提示文件过大

解决方案：使用 Git LFS

# 安装 Git LFS git lfs install # 追踪大文件类型 git lfs track "*.psd" git lfs track "*.zip" git lfs track "*.mp4" # 添加 .gitattributes git add .gitattributes # 正常提交和推送 git add 大文件 git commit -m "添加大文件" git push

7. 权限问题

问题：Permission denied (publickey)

解决方案：

  1. 检查 SSH 密钥是否添加到 GitHub
  2. 测试连接：
    ssh -T git@github.com
  3. 如果失败，重新生成并添加 SSH 密钥（参考"安装配置"章节）
  4. 或使用 HTTPS 方式：
    git remote set-url origin https://github.com/用户名/仓库名.git

8. 清理仓库

问题：仓库体积过大

解决方案：

# 清理不必要的文件和优化仓库 git gc --aggressive --prune=now # 查看仓库大小 git count-objects -vH # 查找大文件 git rev-list --objects --all | \ git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | \ sed -n 's/^blob //p' | \ sort --numeric-sort --key=2 | \ tail -n 10

💡 最佳实践

  • 频繁提交：小步快跑，每完成一个小功能就提交
  • 写好提交信息：清晰描述改动内容和原因
  • 使用分支：不要直接在 main 分支开发
  • 及时同步：每天开始工作前先 pull 最新代码
  • 代码审查：使用 Pull Request 进行代码审查
  • 保护主分支：设置分支保护规则，禁止直接推送
  • 使用 .gitignore：不要提交临时文件、依赖包
  • 定期备份：重要项目定期推送到远程
  • 学习回滚：了解如何撤销错误操作
  • 文档化：维护好 README 和项目文档
